# Events

This page describes the outbox table schema, how Outboxer routes events to a
backend, and the backend-specific options you can attach to an event.

## Event table

The default table and column names are:

```sql
CREATE TABLE events (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    timestamp timestamptz,
    payload text NOT NULL,
    target text,
    destination text,
    options jsonb
);
```

Outboxer reads the table with `SELECT *` and maps columns by name, so optional
columns may simply be left out of the table. Required columns are checked at
startup and a missing one stops the process with a clear error.

| Column | Required when | Notes |
| --- | --- | --- |
| `id` | always | Primary key. Determines event ordering and idempotency, and is used to delete rows. |
| `payload` | always | Message body, passed as-is to the backend. |
| `target` | both backends enabled | Backend selector: `pubsub` or `sqs`. See [Routing](#routing). |
| `destination` | a backend is enabled without a default destination | Pub/Sub topic name or SQS queue URL. Optional when `DEFAULT_PUBSUB_TOPIC` / `DEFAULT_SQS_QUEUE_URL` covers it. |
| `timestamp` | when `MAX_EVENT_AGE_MS` is set | Used for latency logging and age-based poison. |
| `options` | never | Backend-specific JSON options such as ordering keys and attributes. |

A minimal single-backend table can therefore be just `id` and `payload`, with a
default destination configured.

Column names are configurable; see [Configuration](configuration.md#event-table).

Publishing an event is a matter of inserting a row. Outboxer sends the event and
deletes the row once the send is confirmed.

## Routing

Both backends are opt-in. Enable the ones you need with `PUBSUB_ENABLED=true`
and/or `SQS_ENABLED=true`. At least one must be enabled or Outboxer exits at
startup. A backend's client is only created when it is enabled, so disabled
backends never need credentials or configuration.

Routing depends on how many backends are enabled:

- **One backend enabled:** the `target` column is optional and every event is
  sent to that backend.
- **Both backends enabled:** each event's `target` column must be `pubsub` or
  `sqs`, and a configured `target` column (`EVENT_TARGET`) is required, otherwise
  Outboxer exits at startup.

The `destination` resolves to the event's `destination` column, falling back to
`DEFAULT_PUBSUB_TOPIC` / `DEFAULT_SQS_QUEUE_URL` when empty.

Events whose target cannot be routed to an enabled backend (an unknown value, or
an empty target when both backends are enabled), or whose destination is missing
with no default, are logged and left in the table. Fix the row or the
configuration and they are picked up again. These routing failures are **not**
poison and are never deleted or dead-lettered — see [Dead Letter Queue](dlq.md).

## Backend-specific options

The optional `options` column holds a JSON document with a per-backend section.
Outboxer reads the section that matches the resolved target.

### Pub/Sub

```sql
INSERT INTO events (timestamp, payload, target, destination, options)
VALUES (
    now(),
    '{"type":"user.created","id":"123"}',
    'pubsub',
    'user-events',
    '{"pubsub":{"orderingKey":"user-123","attributes":{"source":"users"}}}'
);
```

- `orderingKey` enables ordered publishing for that key. Events that share an
  ordering key are published in `id` order.
- `attributes` are message attributes. Values must be strings; non-string
  values are dropped and logged.

### SQS

```sql
INSERT INTO events (timestamp, payload, target, destination, options)
VALUES (
    now(),
    '{"type":"invoice.created","id":"456"}',
    'sqs',
    'https://sqs.eu-west-1.amazonaws.com/123456789012/invoices',
    '{"sqs":{"attributes":{"source":{"DataType":"String","StringValue":"billing"}}}}'
);
```

- `attributes` use AWS's native `MessageAttributeValue` JSON shape
  (`DataType` + a typed value field).
- `messageGroupId` sets the FIFO message group (FIFO queues only; see below).

#### FIFO queues

Outboxer detects FIFO queues from the `.fifo` suffix on the queue URL, so
standard and FIFO queues are handled correctly without extra configuration:

- **FIFO queues** receive a `MessageGroupId` from `options.sqs.messageGroupId`,
  or a stable synthetic group derived from the event `id` when it has none.
  `MessageDeduplicationId` is also derived from the event `id`. Using the event
  id for deduplication means re-sends after a crash are deduplicated by SQS, so
  content-based deduplication is not required on the queue.
- **Standard queues** never receive a `MessageGroupId` or
  `MessageDeduplicationId`, and batch sends are used.

## Delivery semantics

Outboxer provides at-least-once delivery. A message may be published more than
once if the process stops after publishing but before the database transaction
is committed, or if a queue provider accepts a message but a later operation
fails. **Consumers should be idempotent.**

Outboxer holds a database transaction while publishing a batch. This keeps the
delete behavior simple and preserves ordering, but it also means slow queue
calls can hold row locks for longer. Every database query and publish call is
bounded by a timeout (`PG_QUERY_TIMEOUT_MS`, `PUBLISH_TIMEOUT_MS`) so a single
hung call cannot stall the batch indefinitely, and the watchdog remains as a
last-resort backstop.
