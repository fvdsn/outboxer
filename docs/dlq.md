# Dead Letter Queue

Outboxer can route permanently unsendable events to a PostgreSQL-backed dead
letter queue (DLQ) instead of deleting them. The DLQ is intentionally small and
Outboxer-owned: you configure one table name, and Outboxer writes a JSON
snapshot of the dead-lettered event plus diagnostic metadata.

## Enabling the DLQ

Set `DLQ_TABLE` to a table name to enable the DLQ. An empty value (the default)
disables it. The DLQ table is independent of `EVENT_TABLE` and must not be the
same table.

When `DLQ_TABLE` is set, startup validation verifies that the table exists and
has the required columns; Outboxer never creates it for you.

```sql
CREATE TABLE outboxer_dead_letters (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    event jsonb NOT NULL
);
```

The table has exactly two required columns: `id` and a JSON/JSONB `event`
column. You may add your own indexes or generated columns for querying, but
Outboxer does not depend on them.

## What gets dead-lettered

Only events classified as **content poison** — permanently unsendable as-is —
are dead-lettered:

- payload too large for the backend;
- empty or invalid payload;
- malformed or over-limit attributes/options;
- invalid ordering key or FIFO message group;
- syntactically invalid destination;
- event older than `MAX_EVENT_AGE_MS` (when set).

The guiding rule: if the **unchanged** event could become sendable after
operator action, provider recovery, credential refresh, configuration change, or
an Outboxer upgrade, it is retryable and stays in the outbox table.

### What is *not* dead-lettered

These remain in the outbox table for a later retry and are never dead-lettered:

- **Routing failures** — unknown target, target naming a disabled backend, empty
  target with both backends enabled, or a missing destination with no default.
- **Retryable provider failures** — throttling, timeouts, service unavailable,
  network errors, destination-not-found, permission/credential errors, shutdown,
  or a paused Pub/Sub ordering key.

When the DLQ is **disabled**, poison classification is unchanged — poison events
are simply deleted after classification rather than preserved. Enabling the DLQ
changes *where* poison events go, not *which* events are considered poison.

## Dead-letter payload

The `event` JSON document captures enough context to inspect and manually replay
an event:

```json
{
  "source_schema": "public",
  "source_table": "events",
  "dead_lettered_at": "2026-06-18T12:34:56Z",
  "error": "Event is invalid for SQS",
  "target": "sqs",
  "destination": "https://sqs.eu-west-1.amazonaws.com/123456789012/events",
  "original_event": {
    "id": 123,
    "payload": "{\"type\":\"user.created\"}",
    "target": "sqs",
    "destination": "https://sqs.eu-west-1.amazonaws.com/123456789012/events",
    "options": { "sqs": { "messageGroupId": "user-123" } }
  }
}
```

- `source_schema` — the configured PostgreSQL schema.
- `source_table` — the configured event table name within that schema.
- `dead_lettered_at` — when Outboxer inserted the DLQ row.
- `error` — human-readable diagnostic detail for operators. It is not a stable,
  machine-readable classification API.
- `target` / `destination` — the route Outboxer resolved, filled from routing
  even when the original row left them empty (resolved backend and default
  destination).
- `original_event` — the complete selected event row as Outboxer saw it.

## Transaction semantics

Dead-letter inserts participate in the same batch transaction as confirmed sends
and regular deletes:

- The DLQ insert and the original event delete happen in one transaction.
- If the DLQ insert fails, the original event is not deleted, the batch fails,
  and the database cooldown applies.
- If the transaction commits, the DLQ row exists and the original event is gone;
  if it rolls back, neither is committed (and provider-confirmed sends in the
  batch may duplicate on retry, consistent with at-least-once delivery).

## Replay

Automated replay is out of scope for now. The DLQ payload contains the source
table, resolved route, original row, and error context, which is enough to map
`event.original_event` back into the outbox table manually.
