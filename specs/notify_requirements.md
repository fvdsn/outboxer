# LISTEN/NOTIFY Wake-up Requirements

Status: implementation requirements.

Outboxer should support a PostgreSQL `LISTEN`/`NOTIFY` wake-up so that a process
configured with a poll delay reacts to new events almost immediately instead of
waiting out the full delay. This lowers delivery latency and lets operators
raise the poll interval to spare the database, without sacrificing latency.

The original goal of Outboxer is to be a simpler alternative to Debezium. This
feature keeps that spirit: it is an optimization layered on top of the existing
polling loop, not a new subsystem and not a new operating mode.

## Design Principle: No Modes

There is **no enable/disable flag** for this feature. Behavior is keyed entirely
off the existing `POLL_INTERVAL_MS`:

- `POLL_INTERVAL_MS == 0` (the default): unchanged hot-loop polling. The process
  never sleeps between empty batches, so there is nothing for a notification to
  interrupt. The process does **not** `LISTEN`. This keeps the default behavior
  identical to today — a working default with no surprises for operators who
  never read the docs.
- `POLL_INTERVAL_MS > 0`: between empty batches the process waits for a
  notification but wakes no later than `POLL_INTERVAL_MS`. The interval changes
  meaning from a **latency floor** to a **maximum staleness / safety-sweep
  interval**.

`NOTIFY` is not durable: a notification is lost if no session is listening at the
moment it fires (for example during a reconnect). Therefore the periodic sweep is
the correctness backstop and polling is never removed — `NOTIFY` only
accelerates the wake-up. A missed notification delays an event until the next
sweep; it never loses an event.

## The Trigger Is Operator-Installed and Optional

A notification only fires if something in the database emits it on insert.
Outboxer cannot be woken by an insert it did not make unless that insert causes a
`NOTIFY`. The supported mechanism is a trigger on the outbox table:

```sql
CREATE OR REPLACE FUNCTION outboxer_notify() RETURNS trigger AS $$
BEGIN
  PERFORM pg_notify(TG_ARGV[0], '');
  RETURN NULL;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER outboxer_notify
AFTER INSERT ON events
FOR EACH STATEMENT
EXECUTE FUNCTION outboxer_notify('outboxer_events');
```

Requirements and rationale:

- The trigger is **DDL installed by a provisioning role**, not by the running
  relay. It is either installed by the operator's own migration or generated by
  the `init` command (see [`init_requirements.md`](init_requirements.md)); the
  long-lived relay never issues it.
- Without the trigger, `POLL_INTERVAL_MS > 0` behaves exactly like today's
  polling at that interval. The trigger is a pure latency optimization, never a
  correctness dependency.
- `LISTEN` and `NOTIFY` require **no special database privilege** at runtime, so
  adopting this feature adds **zero** privileges to the Outboxer runtime role,
  which stays least-privilege (DML on the configured tables + `CONNECT`).
- The **running relay** does not auto-create the trigger because that would
  require standing DDL privileges (`TRIGGER` on the outbox table and `CREATE` on
  the schema) on a long-lived service. `CREATE ON SCHEMA` allows arbitrary object
  creation and `TRIGGER` allows attaching code that runs on every application
  insert; both are privilege-escalation surfaces inappropriate for a runtime
  relay. This is consistent with Outboxer's "validate, do not create schema"
  stance for the relay; provisioning DDL lives in the separate `init` command run
  by a privileged role.
- The **channel is passed as a trigger argument** (`TG_ARGV[0]`) and emitted via
  `pg_notify`, rather than hardcoded in the function body. A trigger function is a
  schema-level object shared by name, so a channel baked into the body would mean
  provisioning a second outbox table (re-`CREATE OR REPLACE`-ing the function)
  silently repoints the first table's channel. Keeping the function generic and
  supplying the channel per trigger lets one `outboxer_notify()` serve any number
  of outbox tables. `pg_notify(text, text)` is the function form of `NOTIFY` and
  accepts the channel as a runtime value, which the `NOTIFY` statement cannot.
- The trigger is `AFTER INSERT FOR EACH STATEMENT`, so a bulk insert of many rows
  fires a single notification. The notification carries no payload; a wake-up
  only means "go look", and the existing batch select does the rest.
- `pg_notify` is transactional (identical to `NOTIFY` in that respect): it is
  delivered on `COMMIT`, so the listener is woken only when inserted rows are
  visible.

## Channel Name

- The notification channel name is configurable, defaulting to `outboxer_events`.
- Configuration: `NOTIFY_CHANNEL` env var / `--notify-channel` flag, default
  `outboxer_events`.
- The configured channel must match the channel used by the operator's trigger.
- Only relevant when `POLL_INTERVAL_MS > 0`; ignored otherwise.

## Connection Model

`LISTEN` is connection-scoped: the connection that issues `LISTEN` must be the
same one that waits for notifications. Outboxer keeps a **single database
connection** (`MaxOpenConns(1)`); the listener does not add a second one.

The naive obstacle is the periodic statistics query
(`estimateRemainingEvents`): if it ran in a separate goroutine, a blocking
`WaitForNotification` holding the only connection for up to `POLL_INTERVAL_MS`
would starve it, which would force a second connection. This is avoided by moving
the estimate onto the processor (see [Statistics Estimate](#statistics-estimate))
so nothing queries the database concurrently with the processor.

Requirements:

- The processor loop is sequential and the only database user, so a single
  connection suffices for batch work and the idle wait.
- The listener is **transient**: when `POLL_INTERVAL_MS > 0` and a batch comes
  back empty, the processor borrows the connection (`db.Conn`), issues `LISTEN`,
  waits, and releases the connection before the next batch. When
  `POLL_INTERVAL_MS == 0` no listener is created.
- The wait is implemented via `conn.Raw` to obtain the underlying `*pgx.Conn` and
  `WaitForNotification(ctx)` with a context deadline of `POLL_INTERVAL_MS`. A
  deadline expiry (or parent-context cancellation during shutdown) is a normal
  wake-up; only a genuine connection error is surfaced.
- Borrowing from the pool preserves auto-healing: a dead connection is replaced on
  the next `db.Conn`/`BeginTx`, with no bespoke pinned-connection recovery.
- If the listener cannot be established for an idle cycle, the processor falls
  back to a plain sleep for that cycle and tries again on the next one. The sweep
  timeout covers any notifications missed in between.
- Because the listener only holds the connection while idle, there is a tiny
  window between an empty batch and issuing `LISTEN` in which a notification can
  be missed. Such a row is delivered by the next sweep — acceptable, since the
  sweep is the durability backstop by design.
- When a wait returns because a notification arrived, the listener drains any
  further buffered notifications without blocking so a burst of inserts collapses
  into a single processing cycle.

## Statistics Estimate

The outbox backlog estimate (`events_remaining_estimate`, a cheap
`pg_class.reltuples` read) must not be queried by the statistics logger in its own
goroutine, because that concurrent database access is what would otherwise force a
second connection.

- The processor refreshes the estimate on its own connection, at most once per
  `STATS_INTERVAL_MS`, and caches it.
- The refresh runs **after** a batch completes, never before the next select, so a
  notification wake-up reaches the select without first waiting on the estimate
  query (which could stall up to `PG_QUERY_TIMEOUT`).
- The statistics logger reads the cached value and never touches the database.
- When the estimate is unavailable the field is omitted, as before.

## Watchdog Interaction

- The existing rule that `WATCHDOG_INTERVAL_MS` must be at least 10x
  `POLL_INTERVAL_MS` when polling is enabled is unchanged and still applies.
- Waking on either a notification or the timeout proceeds to a batch cycle that
  marks processor progress, so progress is still marked within the watchdog
  window during idle periods.

## Behavior Matrix

| `POLL_INTERVAL_MS` | Trigger installed | Idle behavior | Idle DB load | Worst-case latency |
| --- | --- | --- | --- | --- |
| `0` (default) | n/a | Hot-loop polling, no `LISTEN` | High (continuous) | ~immediate |
| `> 0` | no | Sweep every interval (today's polling) | Low | up to interval |
| `> 0` | yes | Wake on notify, sweep as backstop | Low | ~immediate |

## Operational Guidance

- With the trigger installed, operators can raise `POLL_INTERVAL_MS`
  significantly to reduce idle database polling while still getting near-immediate
  delivery from the trigger.
- `LISTEN` does not work through connection poolers in transaction-pooling mode
  (for example pgbouncer `pool_mode = transaction`). Outboxer's direct single
  connection is unaffected; this is a documentation caveat for operators who
  front Postgres with such a pooler.

## Multi-Instance & Forward Compatibility

This feature must not constrain future multi-instance parallelization work
(`SKIP LOCKED` claiming, advisory-lock-per-key ordered delivery, a generated
`ordering_key` column). The wake-up is deliberately decoupled from claiming.

- **The notification only means "go look".** Which instance actually gets a row
  is decided entirely by the claim query (`FOR UPDATE SKIP LOCKED` or advisory
  locks), never by the notification. A notification cannot cause a double-send
  regardless of how many instances it wakes. The herd is a performance
  consideration, never a correctness one.
- **All listeners on the shared channel wake on every notification.** This is the
  correct primitive for the work-stealing parallelization model, where any
  instance can claim any key and therefore all instances *should* race for the
  claim. Debounce/drain-fully behavior bounds the cost to N claim queries per
  idle wake-up, not per row.
- **Payload-less is correct, not a limitation.** Plain `LISTEN` cannot filter
  server-side on payload — every listener receives every notification on a
  channel — so a payload would not avoid waking non-owning instances. The only
  lever that avoids the wake is channel partitioning, which the configurable
  channel name keeps open.
- **Per-destination channels are a possible future optimization** for the static
  destination-sharding model (each instance listens only on channels for the
  destinations it owns). This would use a `FOR EACH ROW` trigger calling
  `pg_notify(dynamic_channel, '')`, trading the cheap statement-level single
  notification for per-row routing. It is an additive operator choice in the
  trigger DDL plus multi-channel `LISTEN` on the Outboxer side; it is not a
  redesign. Implementations must therefore not hardcode assumptions that there is
  exactly one channel or that the trigger is statement-level only.
- **The dedicated listener connection does not conflict with the ordered-delivery
  plan.** The decoupled ordered design calls for a session-scoped advisory lock
  held across the publish (auto-released if the instance dies). That lock lives on
  the *processing* connection, which is independent of the listener connection
  added here; the two evolve separately.

## Non-Goals

- No payload-carrying notifications; the wake-up payload is ignored.
- No per-destination or per-target channels; one channel wakes the whole loop.
- No automatic trigger creation by the **running relay**. (The separate `init`
  provisioning command may generate it; see
  [`init_requirements.md`](init_requirements.md).)
- Logical-replication / CDC as an alternative wake source is a separate future
  track, not part of this feature.
- Multi-instance ordered delivery (advisory-lock-per-key, a generated
  `ordering_key` column) is related future work and out of scope here.
