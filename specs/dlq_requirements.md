# Dead Letter Queue Requirements

Status: implementation requirements.

Outboxer should support a PostgreSQL-backed dead letter queue for events that
are permanently unsendable. The DLQ is intentionally small and Outboxer-owned:
operators configure one table name, and Outboxer writes a JSON snapshot of the
dead-lettered event plus diagnostic metadata.

## Configuration

- `DLQ_TABLE` enables the DLQ when non-empty.
- Empty `DLQ_TABLE` disables the DLQ.
- No separate `DLQ_ENABLED` flag is needed.
- The DLQ table name is independent from `EVENT_TABLE`.
- `DLQ_TABLE` must not equal `EVENT_TABLE`.
- When `DLQ_TABLE` is configured, startup validation must verify that the table
  exists and has the required columns.

## Table Shape

The DLQ table has exactly two required columns:

```sql
CREATE TABLE outboxer_dead_letters (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    event jsonb NOT NULL
);
```

- Outboxer does not require or configure any additional DLQ columns.
- Operators may add indexes or generated columns for their own queries, but
  Outboxer must not depend on them.
- Outboxer must not create the DLQ table automatically.

## Dead-Letter Payload

The `event` JSON document must include:

- `source_table`: the configured event table name.
- `dead_lettered_at`: the timestamp when Outboxer inserted the DLQ row.
- `error`: human-readable diagnostic detail when available. This is for
  operators, not a stable machine-readable API.
- `target`: the resolved backend target used by Outboxer.
- `destination`: the resolved destination used by Outboxer.
- `original_event`: the complete selected event row as Outboxer saw it.

The payload must not include a stable machine-readable `reason` field. Error
classification names are internal behavior, not a replay or operator API.

`target` and `destination` must be filled from Outboxer's routing resolution,
even when the original event row did not explicitly contain those values:

- if the target column is absent or empty and exactly one backend is enabled,
  `target` is that enabled backend;
- if the destination column is absent or empty and the routed backend has a
  default destination, `destination` is that default.

Example:

```json
{
  "source_table": "events",
  "dead_lettered_at": "2026-06-18T12:34:56Z",
  "error": "Event is invalid for SQS",
  "target": "sqs",
  "destination": "https://sqs.eu-west-1.amazonaws.com/123456789012/events",
  "original_event": {
    "id": 123,
    "payload": "{\"type\":\"user.created\"}",
    "target": "sqs",
    "destination": "https://sqs.eu-west-1.amazonaws.com/123456789012/events",
    "options": {
      "sqs": {
        "messageGroupId": "user-123",
        "attributes": {
          "source": {
            "DataType": "String",
            "StringValue": "users"
          }
        }
      }
    }
  }
}
```

The JSON shape should be stable enough for operators to inspect and replay
events manually, but it is not a replacement for the source event table schema.

## What Goes To The DLQ

When DLQ is enabled, Outboxer must dead-letter events that are classified as
permanently unsendable content poison:

- P3: payload too large.
- P4: empty or invalid payload.
- P5: malformed or over-limit attributes/options.
- P6: invalid ordering key / message group from backend options.
- P7: syntactically invalid destination.
- P8: event expired by max-age policy.

Routing failures are not poison and must not go to the DLQ:

- R7: target names a disabled backend.
- R10: target unsupported by the current build.
- R11: target ambiguous in the current configuration.
- R12: destination missing and no default configured.

Retryable provider failures must not go to the DLQ:

- R1-R6: provider, network, throttling, destination-not-found, permission, or
  credential failures.
- R8: context canceled / shutdown.
- R9: ordering key paused.

The rule is: if the unchanged event could become sendable after operator action,
provider recovery, credential refresh, configuration change, or Outboxer
upgrade, it is retryable and must remain in the outbox table.

## Transaction Semantics

- Dead-letter insert and original event delete must happen in the same database
  transaction.
- If the DLQ insert fails, Outboxer must not delete the original event.
- If the transaction rolls back, neither the DLQ row nor the original delete is
  committed.
- If the transaction commits, the DLQ row exists and the original event is gone.
- DLQ writes participate in the same batch transaction as confirmed sends and
  regular deletes.

## Disabled DLQ Behavior

When `DLQ_TABLE` is empty, current poison behavior remains:

- confirmed-sent events are deleted;
- content-poison events are deleted after classification;
- retryable and routing failures remain.

The DLQ feature changes where poison events are preserved, not which events are
classified as poison.

## Failure Handling

- DLQ insert failure is a database/transaction failure.
- On DLQ insert failure, the batch must fail and database cooldown applies.
- Sender success must not be undone at the provider, so a later retry may
  duplicate already-confirmed provider sends if the transaction fails.
- DLQ table misconfiguration should fail startup validation when detectable.

## Replay

Replay is out of scope for the first DLQ implementation.

The DLQ payload must contain enough information for manual replay:

- source table name;
- resolved target and destination;
- original event row;
- error context.

A future replay command can map `event.original_event` back into the configured
outbox table.
